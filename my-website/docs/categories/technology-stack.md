# Technology Stack

Feedback relacionado con las tecnologías utilizadas y decisiones técnicas.

---

## ✅ Lo que funciona

- Presentar primero la metodología de desarrollo y posteriormente el stack tecnológico ayuda a contextualizar mejor las decisiones técnicas.
- Justificar claramente las tecnologías elegidas demuestra que las decisiones técnicas están alineadas con las necesidades del proyecto.
- Priorizar APIs, herramientas y tecnologías ampliamente adoptadas facilita la integración, el soporte y la mantenibilidad del sistema.
- Incluir un análisis de riesgos asociado al stack tecnológico permite anticipar posibles problemas técnicos y planificar medidas de mitigación.
- Analizar la curva de aprendizaje de las tecnologías utilizadas ayuda a evaluar el impacto real en la planificación del proyecto.
- Explicar con claridad cómo se integran las diferentes tecnologías dentro del sistema mejora la comprensión de la arquitectura del proyecto.
- Analizar licencias y condiciones de uso de herramientas o librerías open source ayuda a evitar problemas legales o costes inesperados.
- Tener en cuenta la sostenibilidad de créditos, licencias o recursos externos garantiza que las herramientas puedan mantenerse durante todo el desarrollo del proyecto.
- Implementar sistemas de integración continua y aseguramiento de calidad del código desde el inicio contribuye a mantener la estabilidad del proyecto a largo plazo.

---

## ❌ Lo que no funciona

- Presentar el stack tecnológico sin justificar por qué se han elegido esas tecnologías genera dudas sobre las decisiones técnicas del proyecto.
- Elegir tecnologías poco conocidas o poco adoptadas sin un motivo claro puede dificultar la integración y el mantenimiento del sistema.
- No considerar la curva de aprendizaje de varias tecnologías simultáneamente puede afectar negativamente al ritmo de desarrollo.
- No analizar los riesgos asociados a las tecnologías utilizadas puede provocar problemas técnicos imprevistos durante el desarrollo.
- No revisar las condiciones de uso o licencias de herramientas open source puede generar costes o limitaciones inesperadas.
- No considerar la sostenibilidad de créditos o recursos externos puede provocar bloqueos técnicos antes de finalizar el proyecto.
- No explicar cómo se integran las diferentes tecnologías dentro del sistema dificulta entender la arquitectura técnica.
- No establecer desde el inicio mecanismos de aseguramiento de calidad del código o integración continua puede generar problemas acumulados en fases posteriores del desarrollo.
